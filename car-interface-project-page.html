Universal Vehicle Interface
Objective Personas Information Architecture Wireframes Working Prototypes
This is a concept design for a vehicle interface. It was created with the idea of universality in mind; meaning it can be installed in any kind of vehicle: car, truck, etc.


Objective:

To create an interface that drivers can easily use without having to focus on the screen for too long at a time.



Incorporated Pieces:
There were a few different pieces that needed to be considered when designing this interface. The smartphone is a pivotal part as the interface needed to be designed in such a way that it was essentially another smartphone, with the only difference being that the user can access it while driving. The vehicle is essential to the design as the UI needs to be universal across all vehicle platforms. Lastly and arguably most importantly is the user, which the entire interface is designed around.

Research & Analysis
Research was conducted to determine the types of people/personalities that will be using the interface.
Who The Design Is For

This interface was designed entirely around the different kinds of users (drivers) that would be interacting with it. There were three main personalities that were identified to be the most common kind of user.


Kathleen is the first persona that the interface was designed around. She is a busy on-the-go mother of four who is constantly in her car. Between her husband and her kids, she requires full access to her phone to keep in touch with all of them and their obligations. She needs the interface to be simple so that she does not need to constantly think about it while she's thinking about a hundred other things during her busy day.

Grace is a common personality that would use the interface. She's an 18 year old college student who is constantly on her phone, whether it's texting her friends, listening to music, or browsing social media. Due to the risks of texting and driving, Grace is aware that she cannot be on her phone while behind the wheel, so she needs an interface that will allow her to stay in touch all of her friends and listen to her music.

Jim is another personality who the interface was designed for. Jim is a successful businessman who stresses the importance of his time. He doesn't like wasting time learning how to use new things and expects them to be easy to use. He understands the basics of UI in the sense of functionality and practicality and wants an interface that encompasses both.

Wireframes & Flows
These wireframes are essential for the further development of the interface. They were created to provide a map of all the interactions within the application.
Wireframes

Before the final design was constructed, a low fidelity wireframe was created in order to show the flow of the interface and how it is going to move from one page to the other. I have divided the wireframes into common functions that will be used in the interface based on the personas.

Using the Phone Component

This is a low fidelity wireframe of the phone component. The wireframe begins at the main home screen. It then flows to the main phone screen, where the many other phone options are. The first being the standard phone call screen, which allows for regular voice phone calls. Another option is the Phonebook, where the user can select from a grid-like view of circular contact options. From there, they can select which phone number of the contact they'd like to dial.

There are several more options in the phone screen, such as the messaging options. This was specifically designed with the Grace persona in mind and more specifically, so that all users can be safe drivers and keep their hands off their phones while driving.


Using the Audio Component

Another important aspect of the interface is the audio component. From the home screen, the interface flows into the current song that is playing, gives information about it, provides all audio functions such as play, rewind, fast forward, and shows the source of the audio. If the user decides to switch the audio source, he can directly access it from this page by pressing the button in the lower left hand corner. From here, he will be taken to the main sources page in which he can toggle his sources.

The sources page provides easy and simple ways to adjust audio settings. Seeing as majority of all smart phones are connected via Bluetooth, this interface has chosen to make that the primary source input. The user can easily switch between Bluetooth devices by selecting a listed device or by selecting to connect a new one. From here, the new device is paired and brings the user back to the list of sources.


Working Prototypes

After the specific wireframes were put together, a rough prototype was put together to show the working/moving flow of the interface. It was catered to specific cases as listed below.

Audio Case 1: Connect New Bluetooth Device/Add New Source

In this case, the user is connecting an entirely new device to the car. This video brings you through the steps the user will take to connect their new source.


Audio Case 2: Disconnect Existing Device/Connect New Source

This case is a bit different in that the user is disconnecting a device that already exists in the car. This important when looking at personas like Grace, who wish to be able to easily disconnect/reconnect different devices.
*Note that animations are not included in the prototype thus the loading page is not working as intended. That icon would be animated in the true interface.


Phone Case 1: Open New Text Message/Reply with Auto-Reply Feature

In this case, the user is going to be opening a new text message that has arrived in their inbox, and then replying to the message with the auto-reply feature, which has several different reply options that the user can choose from.


Phone Case 2: Write New Text Message/Select Contact/Reply with Auto-Reply Feature

This case is very important in that it enables the driver to interact with their phone while driving, without having their hands physically on it.

The user can select new message from the menu, and then go through a series of movements where he can first (1) select the contact via the phonebook page and then second (2) choose between auto-reply or voice text. Conceptually, in voice text the user will be able to record whatever they desire via voice recognition and then send that message. However, for the sake of this prototype, the user selects auto-reply.
